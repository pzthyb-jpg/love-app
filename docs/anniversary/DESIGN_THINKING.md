# 纪念日模块 — 设计思路

> 版本：v1.0  
> 日期：2026-07-08  
> 目标读者：设计师、开发、产品

---

## 1. 设计原则

### 1.1 极简（Less is More）

- 一个页面只做一件事：管理纪念日
- 不引入图表、不引入时间轴、不引入复杂动画
- 卡片式布局，每张卡片信息密度控制在 4 个字段以内
- 弹窗表单字段 ≤ 5 个

### 1.2 有温度

- 数字后面跟情感文案（「已过 365 天」→「相守一年啦 🎉」）
- emoji 不是装饰，是信息的视觉锚点
- 纪念日当天（_N = 0_）卡片放大 + 金色渐变高亮
- 新增/删除卡片用柔和动画过渡

### 1.3 不打扰

- 推送默认保守：每个纪念日最多 3 次（提前 7 天、3 天、1 天）
- 用户可完全关闭纪念日推送
- 不做每日签到式提醒，不在用户没期待的时候弹出
- App 内不做弹窗广告、不做红点强迫

## 2. 情感目标

> **我们希望用户在打开纪念日 Tab 时，感到的是「被珍视的感觉」，而不是「时间不够用的焦虑」。**

### 2.1 强化仪式感

| 场景 | 设计意图 |
|------|----------|
| 查看列表 | 看到每个纪念日的 emoji + 名称，回忆起「为什么加这个」|
| 纪念日当天 | 卡片高亮 + 特殊文案「今天就是要庆祝的一天」|
| 已相处不断增长的天数 | 数字缓慢递增动画（参考 Home.vue 的 loveDays）|

### 2.2 不制造焦虑

| ❌ 传统设计 | ✅ 我们的设计 |
|------------|-------------|
| 倒计时大红色数字（「还剩 3 天！」） | 柔金色数字（「还有 3 天 💕」）|
| 「又过去了 365 天，你还不好好珍惜？」 | 「相守 365 天啦 🎉」 |
| 错过纪念日弹窗提醒 | 事后温柔提示「昨天是你们的纪念日哦」|

### 2.3 用户情感旅程

```
好奇 → 首次打开 → 看到空态引导 → 添加第一个纪念日
     → 看到自己的纪念日卡片 → 产生「被记录」的满足感
     → 每天来 glance 一下 → 看到数字增长 → 感受到时间的厚度
     → 纪念日收到推送 → 惊喜而非紧张
     → 回望一年 → 数据导出或照片回顾 → 情感沉淀
```

## 3. 设计决策记录

### 3.1 决策 1：纪念日列表的排序方式

| 方案 | 描述 | 取舍 |
|------|------|------|
| A. 按创建顺序 | 用户添加的先后顺序 | ❌ 新旧混在一起，前瞻性差 |
| B. 按原始日期 | 按 date 字段升序 | ❌ 生日在 1 月的永远排前面 |
| C. 按下次发生时间（**采用**） | 自动算出今年下次是哪天，最近的排最前 | ✅ 用户最关心的永远是「下一个」 |

> 实现：计算每个纪念日的 nextOccurrence（今年日期，若已过则 +1 年），按 nextOccurrence 升序排列。

### 3.2 决策 2：纪念日详情的展示方式

| 方案 | 描述 | 取舍 |
|------|------|------|
| A. 独立详情页 | 点击打开 /anniversary/:id | ✅ 更正式，但增加路由和视图 |
| B. 卡片展开（**采用**） | 在当前列表展开 inline 展示详情 | ✅ 更轻量，不离开当前语境 |
| C. 弹窗详情 | 点击弹窗显示完整信息 | ✅ 最轻，但信息容量有限 |

> 选择 B 因为：MVP 阶段一个 Tab 的交互成本最低，且数据量小、信息结构简单。

### 3.3 决策 3：删除交互方式

| 方案 | 描述 | 取舍 |
|------|------|------|
| A. 左滑删除 | 模仿 iOS 邮件滑动删除 | ❌ 移动端兼容性差，Android 用户不习惯 |
| B. 长按弹出删除菜单 | 长按触发 ActionSheet | ❌ 隐藏太深 |
| C. 卡片右上角删除图标 + 确认弹窗（**采用**） | 视觉可见 + 二次确认防误删 | ✅ 明确、安全、跨平台一致 |

### 3.4 决策 4：推送权限的获取时机

| 方案 | 描述 | 取舍 |
|------|------|------|
| A. 首次打开 App | 立刻请求推送权限 | ❌ 用户还没看到价值就要求授权，易拒绝 |
| B. 添加第一个纪念日时 | 保存时弹出请求 | ⚠️ 时机合理但隐蔽 |
| C. 设置页面（**采用**） | 在「推送设置」里显眼位置主动授权 | ✅ 用户心智准备好后才授权 |

### 3.5 决策 5：类型系统的颗粒度

| 方案 | 描述 | 取舍 |
|------|------|------|
| A. 自由文本标签 | 用户自定义标签 | ❌ 信息嘈杂，难做自动映射 |
| B. 单选 type（**采用**） | anniversary/birthday/holiday/custom | ✅ 够用、可视化统一、代码简单 |

## 4. 关键文案规范

| 场景 | 正确文案 | 避免 |
|------|----------|------|
| 纪念日当天 | 今天就是「恋爱纪念日」🎉 | 今天是你的恋爱纪念日！速去庆祝！|
| 倒计时 | 还有 7 天 | 距离纪念日还有 7 天，别忘了！ |
| 已过天数 | 相守 365 天 | 已经在一起 365 天了 |
| 添加成功 | 已添加 💕 | 添加成功！ |
| 空态提示 | 还没有纪念日，添加第一个吧 🎂 | 暂无数据 |
| 删除确认 | 确定要删除【{name}】吗？ | 确认删除？ |
| 推送文案 | 🎂 小宝贝的生日还有 1 天！ | 提醒：您设置的纪念日即将到来 |

> **文案原则**：避免祈使句和感叹号；用 emoji 传递情绪而非标点符号；不用「您」，用「你」或省略。

## 5. 页面层级与心智模型

用户打开 love-app 的导航心智模型：

```
小皮爱情助手（根）
├── 首页（数据看板）
├── 拍照（每日打卡）
├── 午餐（午餐转盘）
├── 愿望（愿望清单）
├── 纪念（纪念日列表）← 新增
└── 设置（全局配置）
```

纪念日模块的自身结构：

```
纪念（Tab 根页面）
├── 列表区
│   ├── 纪念日卡片（重复）
│   │   ├── 卡片内容：emoji、名称、日期、天数
│   │   └── 展开详情区：备注、创建时间、提醒设置
│   └── 空态引导
├── 添加按钮（FAB，右下）
├── 添加/编辑弹窗（Dialog）
└── 删除确认弹窗（Dialog）
```

## 6. 技术选型的设计影响

| 决策 | 设计结果 |
|------|----------|
| 复用 Vant 4 | 用户从其他 Tab 过来不会觉得是「另一个 App」|
| Pinia 响应式 | 添加/删除无需手动刷新，列表即时更新 |
| localStorage 持久化 | 换手机丢数据（预期内，不做云端）|
| 无图片上传 | 界面保持轻量，不引入压缩/裁剪逻辑 |
| CSS 变量体系 | 纪念日主题色可一键切换为 App 全局主题 |
