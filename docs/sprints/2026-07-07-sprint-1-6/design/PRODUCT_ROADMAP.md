## 1. 产品愿景与目标

### 1.1 一句话愿景

> **"让爱情没有门槛——无需登录，无需网络，打开手机就能记录两个人的每一份心动。"**

### 1.2 核心目标（OKR 格式）

| 目标 | 关键结果 | 度量指标 |
|------|---------|---------|
| **O1**: 建立情感化产品信任 | KR1: 数据安全事件归零 | 本地加密覆盖率 100% |
| | KR2: 用户不再担心数据丢失 | 备份成功率 ≥ 99% |
| **O2**: 打造"唯一不可替代"体验 | KR3: AI 彩虹屁使用率 ≥ 80% | 打卡后彩虹屁展示率 |
| | KR4: 午餐转盘成为日均功能 | 日活转盘使用率 ≥ 30% |
| **O3**: 品质达到生产级标准 | KR5: 无障碍合规达标 | WCAG AA 通过率 100% |
| | KR6: 文案风格统一 | 一致性违例数 = 0 |

### 1.3 目标用户优先级

```
┌─────────────────────────────────────────────────────────┐
│  P0: 女朋友（主用户）                                    │
│     └── 每日拍照打卡、查看留言、许愿/吐槽                 │
│                                                         │
│  P1: 男朋友（管理员 / 触发者）                           │
│     └── 管理留言、设置纪念日、查看打卡统计               │
│                                                         │
│  P2: 潜在用户（回流 / 分享接收方）                       │
│     └── 收到打卡卡片、被邀请扫码参与                     │
└─────────────────────────────────────────────────────────┘
```

## 2. 差异化竞争策略

### 2.1 竞争定位矩阵

基于 `COMPETITIVE_ANALYSIS.md` 的竞争格局，love-app 占据 **「轻量 × 隐私 × AI」** 生态位：

```
                    高情感化
                        │
                        │
         恋爱记 ────────┼──────── 小恩爱
          (重社交)      │        (重社交)
                        │
                        │     ★ love-app
                        │     (单机+AI+免登录)
                        │
    低技术壁垒 ─────────┼────────── 高技术壁垒
                        │
                        │
         时光序 ────────┼──────── Between
          (工具化)      │        (设计美学)
                        │
                        │
                    低情感化
```

### 2.2 核心护城河

| 护城河 | 竞品无法复制的原因 | love-app 的壁垒 |
|--------|------------------|----------------|
| **免登录** | 需要后端账号体系的竞品无法回头 | 零门槛即开即用，心智认知强 |
| **纯本地** | 有服务器成本的竞品无法做"纯本地" | 无数据传输，隐私绝对安全 |
| **AI 彩虹屁** | 模板库易复制，AI 生成能力难抄 | 需要 LLM/算法积累 |
| **午餐转盘** | 细分场景无竞品覆盖 | 高频情侣决策场景 |

### 2.3 差异化策略（What We Do vs Competitors）

| 维度 | 竞品做法 | love-app 策略 |
|------|---------|--------------|
| **身份化语言** | Between 用"你/你们"双人称谓语 | 采用男朋友第一人称视角（"宝贝，今天打卡了吗？"） |
| **双人心智** | Paired 每日问题双人回答 | 异步双向——男友预写 + 女友每日收到 |
| **隐私叙事** | 隐私条款藏在设置深处 | 🔒 仅本地存储 · 永不外露（首页可见徽章） |
| **仪式感** | 功能堆砌型 | 每个动作都有情感反馈（彩纸+触感+彩虹屁） |

### 4.2 验收标准

| 检查项 | 标准 | 验证方法 |
|--------|------|---------|
| **密码安全** | 6位以上字母+数字，PBKDF2 哈希+盐值 | 代码审查 + 暴力破解测试（1万次尝试） |

| **长按反馈** | 长按气泡时有进度指示 | 视频录制验证 500ms 进度环 |

## 附录 A: 优先级排序方法说明

本路线图采用 **MoSCoW** 框架划分任务优先级：

| 级别 | 定义 | 本路线图示例 |
|------|------|-------------|
| **Must Have** | 不做就不可发布 / 严重影响核心体验 | 密码安全、Bug 清零、对比度达标、TypeScript |
| **Should Have** | 影响体验但不阻塞发布，可在下个 sprint 补 | 骨架屏、月历视图、PWA 安装提示 |
| **Could Have** | 锦上添花，有空就做 | 折叠屏适配、多语言、Widget |
| **Won't Have** | 明确不做（当前迭代范围内） | Capacitor 打包原生应用、社区功能、SSR |

同时参考 **RICE** 评分的关键要素：

| 因素 | 权重考量 |
|------|---------|
| **Reach (覆盖面)** | 称呼统一 > 仅涉及留言管理 > 修复单个 Bug |
| **Impact (影响面)** | 安全/数据丢失 > UI 细节 > 代码规范 |
| **Confidence (确信度)** | 有明确修复方案 > 需技术预研 > 依赖外部服务 |
| **Effort (工作量)** | 0.5h 快速修复优先安排 > 8h大任务拆分 |
