# 🔄 交叉比对：OpenAI Harness Engineering vs 小皮爱情助手流水线

> 来源：[Harness Engineering: Leveraging Codex in an Agent-First World](https://openai.com/index/harness-engineering/)
> 中文翻译（Notion）：「工程技术：在智能体优先的世界中利用 Codex」
> 比对日期：2026-07-03

## 一、文章原文（中文版）关键段落

> 以下摘要直接取自 Notion 页面 `39237a70-1a56-8046-9437-c54c54aa3946`

### 核心理念

> **「人类掌舵。智能体执行。」**
> — 人类的工作不再是写代码，而是设计环境、明确意图、构建反馈回路

### AGENTS.md 哲学

> **「要给 Codex 的是一张地图，而不是一本 1,000 页的说明书」**
> — AGENTS.md 约 100 行，作为内容目录，指向 docs/ 下更深层的文档
> — 情境是稀缺资源，大指令文件会挤掉任务、代码和相关文档

### 文档树结构

```
docs/
├── design-docs/       (设计文档：核心理念、索引)
├── exec-plans/        (执行计划：active/ + completed/ + tech-debt-tracker)
├── generated/         (自动生成：db-schema.md)
├── product-specs/     (产品规格：onboarding、索引)
├── references/        (参考：设计系统、nixpacks、uv 的 LLM 上下文)
├── DESIGN.md          (设计方案)
├── FRONTEND.md        (前端架构)
├── PLANS.md           (计划总览)
├── PRODUCT_SENSE.md   (产品感)
├── QUALITY_SCORE.md   (质量评分，每个领域 + 趋势追踪)
├── RELIABILITY.md     (可靠性)
└── SECURITY.md        (安全)
```

### 可读性目标

> **「从智能体的角度来看，它在运行时无法在情境中访问的任何内容都是不存在的。」**
> — 代码优化首先为 Codex 的可读性，其次才是人类
> — Google Docs、聊天记录、人脑中的知识 → 智能体无法访问
> — 仓库本地、已版本化的工件（代码、Markdown、可执行计划）才是智能体能看到的全部

### 架构强制

> **「通过强制执行不变量，而非对实施过程进行微观管理」**
> — 分层架构：Types → Config → Repo → Service → Runtime → UI
> — 横切关注点通过单一 Provider 接口
> — 自定义 linter + 结构测试机械强制执行
> — lint 错误信息直接注入修复指令到智能体上下文

### 黄金原则 + 垃圾回收

> **「Codex 会复现代码库中已存在的模式 — 即使是不均衡或不理想的」**
> — 最初每周五花 20% 时间清理「AI 残渣」
> — 后来改为：黄金原则编码到仓库 + 定期后台任务扫描偏差 + 自动开重构 PR
> — 大多数 PR 一分钟内审完自动合并

### 吞吐量哲学

> **「纠错成本低，而等待成本高」**
> — 减少阻塞合并门，PR 生命周期短
> — 测试偶发失败后续重跑，不无限期阻塞

## 二、逐项深度比对

### 比对 1：AGENTS.md — 目录 vs 百科全书

| 维度 | OpenAI | 小皮助手 | 差距 |
|------|--------|---------|------|
| 大小 | ~100 行纯目录 | ~60 行含项目结构+功能描述+架构约束 | ⚠️ 我们仍含详细描述 |
| 样式 | 纯地图+指针，无详细内容 | 混合了目录和内容 | ⚠️ 可精简 |
| 指向 | 指向 design-docs/exec-plans/product-specs/ 等 | 指向 docs/ 下平铺文件 | ⚠️ docs 未分层 |
| 新鲜度 | CI 验证知识库更新/交叉链接 | 依赖 watchdog 每日扫描 | ✅ 有机制 |

**改进：** AGENTS.md 精简到纯目录格式，详细内容移动到子目录

### 比对 3：代理可读性 — 思维模式差异

| 维度 | OpenAI | 小皮助手 |
|------|--------|---------|
| **代码优化目标** | 优先让代理推理，其次人类 | 优先人类可读 |
| **文档存储位置** | 全在 repo 内，可版本化 | 全在 repo 内 | ✅ |
| **Slack/讨论内容** | 必须沉淀到 repo，否则代理看不见 | 少量沉淀到 LESSONS.md | ⚠️ 不彻底 |
| **依赖选择** | 偏好「无聊」技术（稳定、API 稳定、训练集中常见） | Vue 3 + Vite（主流） | ✅ |
| **边界校验** | 在边界解析+验证数据形状，不猜测类型 | safeGetJSON + PRD 字段定义 | ✅ |
| **重实现 vs 引入库** | 宁可重实现子集也不引入黑盒依赖 | 零依赖策略 | ✅ |

**核心差距：** 我们还没有把「这个信息智能体能不能读到？」作为设计决策的默认问题。

---

### 比对 4：架构强制 — 机械 vs 文档

| 维度 | OpenAI | 小皮助手 |
|------|--------|---------|
| 分层规则 | Types→Config→Repo→Service→Runtime→UI（linter 强制执行） | 无显式分层规则 | ❌ |
| 依赖方向 | 只允许前向依赖，横切关注点通过 Provider | Vue 单向数据流（父→子） | ⚠️ 部分有 |
| 自定义 linter | ✅ 代码生成 + CI 阻断 | ❌ 无 | ❌ |
| 文件大小限制 | linter 强制执行 | 无约束 | ❌ |
| 命名规范 | linter 强制执行 | README 中约定 | ⚠️ 未机械执行 |
| 结构化日志 | linter 强制执行 | 未统一日志 | ❌ |

**核心差距：** 我们靠文档约定，OpenAI 靠机械强制执行。

### 比对 5：自验证能力

| 维度 | OpenAI | 小皮助手 |
|------|--------|---------|
| 构建验证 | ✅ | ✅ `npm run build` |
| 运行时验证 | ✅ 启动 app + Chrome DevTools + 截图 + DOM 快照 | ❌ 无 |
| Bug 复现 | ✅ 自动录制视频演示故障 | ❌ 无 |
| 验证修复 | ✅ 再次运行 app 验证 + 录解决视频 | ❌ 无 |
| 可观测性 | ✅ LogQL + PromQL 查询 | ❌ 无 |
| 持续运行 | ✅ 单次运行 6+ 小时（人睡觉时） | ⏳ 子代理 10-30 分钟 |

**核心差距：** 我们没有让代理能「看到」运行时的 app。

---

### 比对 6：审核模式

| 维度 | OpenAI | 小皮助手 |
|------|--------|---------|
| 审核主体 | 智能体对智能体（Ralph Wiggum 循环） | 审核智能体 → 主代理中转 | ⚠️ |
| 人类介入 | 仅在需要判断时 | 每次审核结果需主代理确认 | ⚠️ 可减少 |
| P2 处理 | 不阻塞，自动合并 | 触发 code_redo 或记录 | ⚠️ 可优化 |
| 审核者自愈 | ✅ 代理自己拉取反馈、内联回复、推送更新 | 审核产出 → 主代理重派 | ⚠️ 需中转 |

**核心差距：** 审核智能体应能直接修复 P2 问题并 commit，不经过主代理。

---

### 比对 7：质量追踪

| 维度 | OpenAI | 小皮助手 |
|------|--------|---------|
| 质量评分 | QUALITY_SCORE.md（每个领域+趋势） | ❌ 无 |
| 技术债务 | tech-debt-tracker.md | LESSONS.md（教训记录） | ⚠️ 有但非结构化 |
| 垃圾回收 | 定期后台任务扫描偏差+开重构 PR | watchdog 仅查 docs vs 代码 | ⚠️ 范围窄 |
| 人类品味编码 | 审查评论→文档→编码到工具 | 手动记录到 LESSONS.md | ⚠️ 未自动执行 |

## 三、差距优先级矩阵

| 优先级 | 差距 | 影响 | OpenAI 做法 | 预估 |
|--------|------|------|-----------|------|
| **P0** | 无运行时自验证 | Bug 逃逸到运行时才发现 | 启动 app + DevTools + 截图验证 | 2h |
| **P0** | 无自定义 linter/架构强制 | 架构随迭代退化 | 分层规则 + linter + CI 阻断 | 4h |
| **P1** | docs/ 未分层，exec-plans/ 缺失 | 执行计划无存档，信息查找低效 | design-docs/ + exec-plans/ + quality/ | 2h |
| **P1** | 无 QUALITY_SCORE.md | 无法量化代码质量趋势 | 每领域评分 + 趋势追踪 | 1h |
| **P1** | watchdog 范围太窄 | 代码品味退化不被发现 | 扫描代码质量/架构偏差+开重构 PR | 3h |
| **P1** | 审核智能体不能直接修复 P2 | 主代理成为瓶颈 | 代理间审核，P2 自动修复合并 | 1h |
| **P2** | AGENTS.md 含详细描述（非纯目录） | 渐进披露不彻底 | ~100 行纯目录+指针 | 0.5h |
| **P2** | 无自愈机制（构建失败需人工重试） | 子代理效率损失 | 自动分析+重试，N 次失败才报人 | 2h |
| **P2** | 人类品味未编码为机械规则 | 同类问题反复出现 | 审查评论→文档→linter→自动执行 | 3h |

## 四、OpenAI 最值得学习的 3 个认知

### 认知 1：代理可读性 > 人类可读性

> **「从智能体的角度来看，它在运行时无法在情境中访问的任何内容都是不存在的。」**

这应该成为我们每次设计决策的默认问题：
- 这个信息智能体能读到吗？
- 这个依赖智能体能推理吗？
- 这个架构智能体能导航吗？

### 认知 2：机械约束 > 文档约束

> **「通过强制执行不变量，而非对实施过程进行微观管理」**

文档会腐烂，linter 不会。我们应该把更多规则从 .md 移到 .js 里：
- Vue 组件大小检查 → 添加到 lint 脚本
- CSS 变量使用检查 → 添加到 lint 脚本
- 导入方向检查 → 添加到 lint 脚本

### 认知 3：垃圾回收是基础设施，不是额外工作

> **「技术债务就像一笔高息贷款：不断地以小额偿还，总比让债务累积再痛苦地一次解决要好。」**

Watchdog 不应只查 docs vs 代码，还应该：
- 扫描代码质量偏差
- 检查架构规则违反
- 自动开修复 PR
- 每日运行，每 PR 一分钟内审完

## 五、总结

| 维度 | OpenAI 评分 | 小皮助手评分 |
|------|-----------|------------|
| 多代理角色分工 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ ✅ |
| 反馈闭环协议 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ ✅ |
| 检查点/容错 | ⭐⭐⭐ | ⭐⭐⭐⭐ ✅ |
| 安全审核 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ ✅ |
| 代理间审核 | ⭐⭐⭐⭐⭐ | ⭐⭐ ⚠️ |
| 架构机械约束 | ⭐⭐⭐⭐⭐ | ⭐ ❌ |
| 运行时自验证 | ⭐⭐⭐⭐⭐ | ⭐ ❌ |
| 质量追踪 | ⭐⭐⭐⭐ | ⭐⭐ ⚠️ |
| 文档分层 | ⭐⭐⭐⭐⭐ | ⭐⭐ ⚠️ |
| 垃圾回收范围 | ⭐⭐⭐⭐⭐ | ⭐⭐ ⚠️ |

**一句话总结：** 我们的**流程骨架**（谁做什么、怎么反馈、怎么容错）已经对标得不错了。差距主要在**机械层面**（linter、自验证、架构强制）和**代理体验设计**（代理可读性、代理间审核、质量追踪）。
