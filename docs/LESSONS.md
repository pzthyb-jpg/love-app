## 一、常见文档-代码不一致模式

### [虚构 API] — 文档声称依赖某外部 API，但代码使用预设数据
- **表现**：AGENTS.md 声称「高德地图 API」驱动餐厅功能，实际代码使用 10 家预设虚拟餐厅，无任何高德接入
- **根因**：设计方案阶段高估了技术能力边界，未验证 API 可行性和密钥安全性就直接写入文档
- **预防措施**：每个 API 依赖在写入文档前必须经过「可行性验证」（API Key 是否已申请？纯前端是否能安全使用？）。文档中的「技术栈」部分应与代码实际使用的技术严格对应

### [夸大数量] — 夸大引擎生成能力
- **表现**：AGENTS.md 声称彩虹屁引擎有「1000+ 模板变体」，实际代码只有约 30 个基础模板（尽管通过随机组合可以产生多倍变体，但 1000+ 的声称不准确）
- **根因**：为增强项目吸引力进行了数字夸大，但后期未核实（可能是 ChatGPT 自动生成的夸张描述）
- **预防措施**：所有量化声称必须从实际代码中统计得出。在 AGENTS.md 中标注「模板数 = 基础模板数 × 变体组合数」的计算公式

### [虚构文件结构] — 文档目录与代码实际结构不符
- **表现**：AGENTS.md 声称存在 `assets/icons/` 目录，实际不存在；项目实际使用纯 emoji 而非图标文件
- **根因**：文档先行编写时未与代码进行对齐检查，引入了设计阶段预期但未实现的文件结构
- **预防措施**：每次提交前运行 `scripts/validate-docs.sh`（或手动检查）对比文档中的文件路径与实际文件系统

### [功能承诺未实现] — 文档描述与实际行为不一致
- **表现**：DESIGN.md 描述拍照流程为 4 状态（含「用这张」确认步骤），但 PRD.md 和实际代码为 3 状态流程（自动保存无确认）
- **根因**：设计方案撰写后，迭代过程中做出了简化决策但未同步更新设计文档
- **预防措施**：建立「设计变更通知」机制——任何从方案到实现过程中的简化/调整必须同步更新 DESIGN.md

### [功能优先级不匹配] — 设计文档与 PRD 优先级矩阵不同
- **表现**：DESIGN.md 将「数据备份导出」列为 P1 并建议升级为 P0，但 PRD.md 完全未将其纳入功能清单
- **根因**：阶段一的审核建议未在 PRD 中体现，缺少「审核反馈→文档更新」的闭环流程
- **预防措施**：在 PRD 模板中增加「审核反馈采纳说明」章节，强制记录每条审核建议的处理结果

## 二、设计阶段容易遗漏的问题

### [男友管理端入口] — 男朋友「虚拟在场」的关键基础设施
- **问题**：F-03 留言板要求「男朋友预写入留言」，但整个设计方案没有任何关于「男朋友用什么界面写留言」的交互定义
- **根因**：设计焦点完全在「女友端的展示效果」（首页预览、留言板详情），忽略了「输入端」的设计
- **教训**：任何「双人」功能必须同时定义 A 端的写入方式和 B 端的读取方式。不要假设「男朋友自然会知道怎么写入」。设计阶段必须产出双方的完整交互流程
- **解决方案**：推荐隐藏管理端入口（连续点击标题 5 次或 `?admin=true`），提供轻量 CRUD 界面

### [iOS localStorage 容量限制] — 移动端存储瓶颈
- **问题**：base64 照片存储 14 天约 2.1MB，iOS Safari localStorage 上限约 5MB。当前设计未考虑照片膨胀风险（如果每天拍多张或照片质量过高，14 天可能超限）
- **根因**：设计阶段按「理想条件」估算存储（单张 ~150KB × 14 天 = 2.1MB），未考虑最坏情况（单张 >200KB × 每天拍多张 × 14 天 > 5MB）
- **教训**：所有容量/性能预估必须同时给出「理想值」和「最坏值」。移动端 localStorage 上限不是「约 5MB」而是「iOS 上就是 5MB，无任何余量」
- **解决方案**：存储上限预留 50% 安全余量。建议改用 IndexedDB（iOS 无痕模式下可用，约 50MB+ 上限）或使用 `safeSetJSON` 中的 `autoCleanupStorage()` 自动清理

### [页面导航的「从哪里来，到哪里去」闭环] — 返回路径缺失
- **问题**：DESIGN.md 设计了多个新页面（全屏浏览、留言板详情、设置页），但仅在 DESIGN_THINKING.md 中提到「统一使用左上角返回按钮 + 右滑返回」，PRD 中完全无返回路径设计
- **根因**：每个新功能独立设计时，只关注「进入」的交互，未设计「退出」的交互
- **教训**：每个页面设计必须包含 4 个要素：入口、操作、出口、降级路径。缺少任何一个，该功能的交互设计就不完整
- **解决方案**：PRD 中每个功能增加「导航路径」小节：从哪里来 → 用户做什么 → 到哪里去 → 回不去了怎么办

### [留言循环递归风险的遗漏] — 伪代码中的隐藏 Bug
- **问题**：PRD §4.3 的 `getTodaysMessage()` 包含未保护的递归调用。当所有留言都已展示后，虽重置 `displayedDates` 但递归重新执行可能再次匹配失败导致无限递归
- **根因**：设计阶段使用的「伪代码」没有经过编译器/静态分析，无法暴露运行时风险和终止性问题
- **教训**：PRD 中的代码段应被视为可运行代码来审查。特别注意递归/循环的终止条件、异步操作的竞态条件、回调的未执行路径
- **解决方案**：加安全阀 `if (retryCount > MAX_RETRY) return fallbackMessage;`

### [用户性别视角偏差] — 「男朋友」在留言板中的被动角色
- **问题**：所有功能的交互设计都围绕「女友的体验」展开，男朋友被默认为「被动的留言写入者」。没有任何功能设计「男朋友的体验」——他看不到她的打卡、看不到她的愿望进展、不知道她的打卡数据
- **根因**：产品定位是「送个女朋友的工具」，设计思维局限在「一个人使用」，未意识到「两个人」的体验都应被设计
- **教训**：任何双人/情侣产品，必须分别列出两个人的使用场景和情感旅程。如果只有一个人的场景被设计，最终产品就是「一个人的工具」
- **解决方案**：在 DESIGN_THINKING.md 中增加「他的视角」一章，画出男朋友使用 App 的旅程地图

### [新用户引导缺失] — 首次用户体验断层
- **问题**：PRD 没有任何关于「第一次打开 App」的流程设计。新用户面临：无纪念日、无打卡记录、无留言、无愿望——首页可能是空的，无法引导用户完成首次体验
- **根因**：设计迭代聚焦在已有用户的功能增强，忽略了「新用户」这个重要角色
- **教训**：每次功能迭代都应该问「新用户打开 App 的第一屏是什么样子的？」如果新增功能依赖「已有数据」才能展示，就必须设计无数据时的引导流程
- **解决方案**：首页检测 `love_anniversary` 是否存在，若无则弹出设置向导（设置纪念日 → 打卡自拍 → 逛逛愿望池 → 转转午餐盘——一次引导完成核心功能的首次体验）

## 三、可复用的最佳实践

### [safeGetJSON / safeSetJSON 工具函数模式] — 存储安全的核心模式
- **价值**：⭐⭐⭐⭐⭐ 整个项目的存储安全性基石
- **核心要点**：
  1. 所有 `JSON.parse()` 必须包裹在 try-catch 中
  2. 数据损坏时自动修复（清除 key + 返回默认值）
  3. 写入时捕获 `QuotaExceededError` 并触发自动清理
  4. 写入失败返回 boolean，供上层决定是否向用户反馈
- **可复用性**：任何使用 localStorage 的前端项目都应实现此工具函数对

### [单向数据流模式] — 纯前端 SPA 的数据管理方案
- **价值**：⭐⭐⭐⭐ 无框架应用的合适架构
- **核心要点**：
  1. localStorage 是唯一数据源（SSOT）
  2. 页面渲染时从 localStorage 读取并聚合
  3. 操作写入时直接更新 localStorage
  4. 首页作为唯一的数据聚合层（读取所有 key 并统计）
- **可复用性**：适用于所有「纯前端 + localStorage」的小型 SPA 项目

### [calculateStreak 连续打卡算法] — 通用的 streak 计算模式
- **价值**：⭐⭐⭐⭐ 从 Duolingo 借鉴的核心算法
- **核心要点**：
  1. 从今天开始逐日向前检查
  2. 同一天多条记录只算一天
  3. 今天没打卡但昨天有打卡 = 连续（今天还没到截止时间）
  4. 前序记录中有空白日期 = 中断
- **可复用性**：任何包含「连续打卡」功能的应用

### [成就里程碑 + Streak 混合模型] — Duolingo 式习惯养成
- **价值**：⭐⭐⭐⭐⭐ 提升留存的最有效模式
- **核心要点**：
  1. Streak 数字提供每日即时反馈（打卡后数字+1 动画）
  2. 里程碑（7/14/21/30 天）提供阶段性目标和庆祝仪式
  3. 已获得的徽章永久保留（不可磨灭的成就）
  4. 进度条显示「下一个里程碑还差 X 天」
- **可复用性**：打卡、学习、健身等所有习惯养成类应用

### [状态机驱动的 UI 流程设计] — 简化复杂交互流程
- **价值**：⭐⭐⭐⭐ 拍照流程重构的核心方法论
- **核心要点**：
  1. 将流程中所有可能状态穷举（IDLE / CAMERA_OPENING / CAMERA_READY / PHOTO_CAPTURED / ERROR）
  2. 每个状态定义：显示什么 UI、哪些按钮可用、触发什么事件
  3. 状态转移用箭头明确标明
  4. 每个转移路径有对应的兜底处理
- **可复用性**：任何包含多步骤交互的功能（拍照、支付、设置向导等）

### [Easter Egg 隐藏入口 + 轻量认证] — 管理端安全方案
- **价值**：⭐⭐⭐⭐ 单机版应用的管理端鉴权好方案
- **核心要点**：
  1. 入口隐藏（连击标题 5 次 / 特殊 URL 参数）
  2. 简单的数字密码鉴权（4-6 位，存 localStorage）
  3. 避免引入全栈认证系统
  4. 密码仅用于防误触，不做高强度安全要求
- **可复用性**：所有单机版 / 纯前端应用的管理功能

### [CSS 变量统一设计语言] — 零成本全局重构
- **价值**：⭐⭐⭐⭐⭐ 改动最小、影响最大的设计技术
- **核心要点**：
  1. 所有颜色、圆角、间距、阴影使用 CSS 变量
  2. 设计变更只需改变量值，无需搜索替换
  3. 同一套变量可以被多个主题复用
  4. ZA Bank → 温暖恋人版的风格升级只需改 20 个变量值
- **可复用性**：所有前端项目

## 四、跨项目通用的教训

### [模式 1：数据孤岛] — 功能模块之间的信息断裂
- **表现**：三个功能（拍照、午餐、愿望）各自独立，在 v1.0 中没有「跨功能」的数据展示。女友的愿望男朋友看不到、午餐记录和打卡记录无关联
- **根因**：功能从独立模块逐步叠加，缺乏「产品整体信息架构」的顶层设计
- **预防措施**：每次新增功能时先画信息架构图，标注「新功能与已有功能的数据连接点」。首页仪表盘就是解决这个问题的设计模式

### [模式 2：输入端设计缺失] — 只设计消费端，不设计生产端
- **表现**：F-03 留言板设计了女生的查看体验（首页预览、留言板详情），但完全没有设计男朋友写入留言的界面
- **根因**：产品经理/设计师自然默认「她」为最终用户，忽略「他」也是用户
- **预防措施**：每个功能的需求文档中必须有「数据从哪里来」和「数据到哪里去」两个表格。如果数据来自其他用户，必须设计该用户的输入界面

### [模式 3：最坏情况不做预估] — 容量/性能只算理想值
- **表现**：存储预估只算平均场景（2.1MB），未计算最坏场景（每天拍多张 + 大文件 = 可能超 5MB 上限）
- **根因**：设计阶段的估算天然倾向于「乐观偏差」
- **预防措施**：所有的容量/性能/时间预估必须同时给出：理想值、预期值、最坏值。最坏值决定了系统是否需要兜底策略

### [模式 4：无数据状态的设计盲区] — 空状态被忽视
- **表现**：多个功能（首页、照片墙、成就系统）的设计集中在「有数据时的体验」，但「第一次打开 App 时」和「数据被清除后」的体验几乎未设计
- **根因**：迭代设计时默认「已有用户 + 已有数据」，忘记了新用户和故障恢复场景
- **预防措施**：每个功能的验收标准必须包含「空状态」验收项。检查清单：首次打开/数据被清除/所有数据已展示完/网络离线（如果依赖网络）

### [模式 5：审核建议无落地跟踪] — 审核变成「一次性的评论」
- **表现**：阶段一审核提出了 7 项需求遗漏、5 项边界情况，但在 PRD 中只有部分被采纳（如留言管理端入口被设计但入口方式与设计不一致）
- **根因**：审核报告撰写后「交给产品经理自己决定是否采纳」，没有闭环跟踪机制
- **预防措施**：建立审核建议跟踪表：每条建议有 ID、状态（未处理/处理中/已采纳/已拒绝）、负责人、截止日期、最终结果。每个里程碑回顾时检查

### [模式 6：跨文档一致性无自动检查] — 多份文档各自为政
- **表现**：AGENTS.md 声称 1000+ 模板和高德地图，DESIGN.md 说 7 天照片墙，PRD.md 说 14 天。三份文档中的数字和事实不一致
- **根因**：没有「跨文档校验」机制，每个文档独立版本管理，修改一个不同步其他
- **预防措施**：
  1. 建立 `docs/ALIGNMENT_CHECKLIST.md`，列出所有跨文档共享的关键数字和事实
  2. 每次提交前运行脚本检查这些事实在各文档中的一致性
  3. 考虑使用文档模板中的「引用字段」而非手打数字（如 `{{PHOTO_WALL_DAYS}}`）

### [模式 7：纯前端的「安全」幻觉] — 隐藏入口 ≠ 安全入口
- **表现**：留言管理端通过「连续点击标题 5 次」隐藏入口 + 「4 位数字密码」保护。但密码保存在 localStorage 中（明文），只要懂一点 Web 调试就能绕过
- **根因**：将「防误触」等同于「安全保护」，低估了单机版应用的隐私风险
- **预防措施**：明确区分「防误触」（UI 层面的隐藏）和「安全保护」（需要真正的加密/后端验证）。对于敏感数据（如两人的私密留言），纯前端的任何「安全」措施都是防君子不防小人
