## 三、反馈闭环协议

### 3.1 反馈消息格式

当审核智能体发现问题需要回溯时，输出格式如下：

```
[FEEDBACK] 类型: design_redo / code_redo
[FEEDBACK] 严重程度: P0/P1/P2
[FEEDBACK] 目标: [具体文件路径]
[FEEDBACK] 问题: [问题描述]
[FEEDBACK] 建议方案: [修复建议]
[FEEDBACK] 上下文: [相关代码/文档引用]
---
```

| 类型 | 含义 | 接收方 | 执行动作 |
|------|------|--------|----------|
| `design_redo` | 设计/需求层面有问题 | 🎨 设计智能体 | 带着反馈列表重新执行设计阶段 |
| `code_redo` | 代码实现不符合设计或有 Bug | 💻 代码智能体 | 带着反馈列表重新执行代码阶段 |

### 3.2 反馈触发条件

#### 触发 design_redo 的场景
- 字段定义不完整或缺失（P0/P1）
- 数据流闭环存在缺口
- 交互设计方案不可实现
- 功能优先级划分明显不合理
- PRD 与 DESIGN 存在重大不一致
- 遗漏了关键功能或极端情况

#### 触发 code_redo 的场景
- 代码实现了与设计/PRD 不一致的行为
- 功能逻辑有 Bug（边界条件未处理）
- 性能问题（如 base64 照片存储超限）
- 交互设计与 DESIGN.md 的 ASCII 线框图不符
- 代码引入了安全/隐私风险
- 未按 PRD 定义的字段结构实现数据层

### 3.3 降级处理（非阻塞反馈）

对于 P2 级别的问题（锦上添花/风格建议），不需要触发重新执行，而是：
- 追加到 `docs/REVIEW_DESIGN.md` 的对应阶段章节
- 标记为 `[OPTIONAL]`，供后续迭代参考
- 不阻塞当前流程

## 五、使用方式

当审核智能体发现问题需要重新执行：

1. **触发反馈**：在 REVIEW_DESIGN.md 中按 `[FEEDBACK]` 格式输出问题
2. **记录追踪**：问题同步记录到 LESSONS.md 的问题模式
3. **重派任务**：主智能体（Hermes）读取 FEEDBACK，重新 dispatch 设计或代码智能体，附带上问题列表作为 context
4. **验证闭环**：修正后的产出再次进入审核——形成闭环

示例：
```
→ 🕵️ 发现代码中照片存储用了 localStorage 而非 PRD 要求的 IndexedDB
→ 输出 [FEEDBACK] type: code_redo, severity: P0
→ Hermes 读取后重新 dispatch 💻 代码智能体，带上 "照片存储需改用 IndexedDB"
→ 💻 代码修正后 commit
→ 🕵️ 再次审核验证 → ✅ 通过
```

## 六、容错与断点续跑协议

### 6.1 状态追踪文件

**`CHECKPOINT.md`** 是单文件真相源，记录了：
- 整体流水线进度百分比
- 每个阶段的完成情况
- 每个代码步骤的完成状态 + commit hash
- 下一步该做什么

### 6.2 重启恢复流程

当中断后重新打开会话：

```
Step 1: 读取 memory → 找到 CHECKPOINT.md 路径
Step 2: 读取 CHECKPOINT.md → 确认当前进度
Step 3: 读取 git log → 确认最新 commit
Step 4: 根据 CHECKPOINT.md 的「下一步动作」继续
```

### 6.3 状态更新规则

- 每完成一个步骤 → 更新 CHECKPOINT.md 并 git commit
- 每次 dispatch 子代理 → 在 CHECKPOINT.md 标记 "进行中"
- 每次子代理返回 → 更新对应步骤为 "已完成" + 记录 commit hash
- 中断恢复时 → 先读 CHECKPOINT.md，再读 git log 交叉验证

### 6.4 冲突处理

如果两个子代理同时工作导致冲突：
1. 以最新 commit 的版本为准
2. 有冲突的文件，取后提交的版本
3. 更新 CHECKPOINT.md 标记冲突文件需重新审核
