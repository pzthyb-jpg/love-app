## 1. 交互流程审计

### 1.1 拍照打卡 (Photo.vue)

**主流程**: 打开页面 → 点击「咔嚓拍照」→ 授权摄像头 → 拍照 → 确认/重拍 → 保存

| 问题编号 | 严重度 | 问题描述 | 代码位置 |
|---------|--------|---------|---------|
| UX-P-001 | 🔴 高 | 摄像头权限被拒绝时仅 Toast 提示，无引导用户前往系统设置的入口 | `Photo.vue:415` `showToast({ message: '😢 无法打开摄像头，请检查权限设置', type: 'fail' })` |
| UX-P-002 | 🟡 中 | 拍照后直接进入「已确认」状态，缺少预览确认环节，用户可能因表情/构图不满意但已无法撤回 | `Photo.vue:443-484` `confirmPhoto()` 直接保存 |
| UX-P-003 | 🟡 中 | 提醒设置与 Settings.vue 完全重复，造成用户困惑应在何处修改 | `Photo.vue:59-86` vs `Settings.vue:84-126` |
| UX-P-004 | 🟢 低 | 照片墙空状态判断逻辑有冗余：`v-if="recentPhotos.length > 0"` 内部又嵌套 `v-if="recentPhotos.length === 0"` | `Photo.vue:101-113` |
| UX-P-005 | 🟡 中 | 成就徽章进度条使用 `nextMilestone.progress` 但未做 0-1 范围保护，可能因浮点精度显示异常 | `Photo.vue:138` |

**流程断点分析**:
```
[用户点击拍照] → [权限请求] → ❌ 拒绝 → Toast 提示 → ❌ 无后续操作指引
                                    ↓
                              ✅ 授权 → [打开摄像头] → ❌ 失败 → Toast 提示 → ❌ 无重试按钮
```

### 1.2 午餐转盘 (Lunch.vue + LunchWheel.vue)

**主流程**: 查看转盘 → 点击中间按钮 → 等待 4 秒 → 查看结果 → 确认/重转/导航/收藏

| 问题编号 | 严重度 | 问题描述 | 代码位置 |
|---------|--------|---------|---------|
| UX-L-001 | 🟡 中 | 转盘旋转期间按钮 `disabled` 但无倒计时或进度提示，用户不确定是否生效 | `LunchWheel.vue:26-30` `:disabled="isSpinning"` |
| UX-L-002 | 🟡 中 | 「再转（排除当前）」操作不可逆，无确认提示，用户可能误触 | `Lunch.vue:223-232` `spinAgainExclude()` |
| UX-L-003 | 🟢 低 | 餐厅管理折叠面板默认展开 (`showManage = 'manage'`)，与「折叠」的交互预期不符 | `Lunch.vue:132` |
| UX-L-004 | 🟡 中 | 添加餐厅时 emoji 随机分配，用户无法自定义，可能与实际餐厅类型不匹配 | `Lunch.vue:262-263` |
| UX-L-005 | 🟢 低 | 导航功能使用高德地图 URI Scheme，但未检测用户是否安装高德地图 | `Lunch.vue:238-239` |
| UX-L-006 | 🟡 中 | 转盘结果弹窗中「好！去吃！」按钮关闭弹窗但无实际功能（如记录选择） | `Lunch.vue:218-221` `confirmResult()` 仅关闭弹窗 |

**流程断点分析**:
```
[点击转盘] → [旋转 4s] → ❌ 期间用户可点击其他 Tab → 结果弹窗被遮挡
                    ↓
              [结果弹窗] → [导航] → ❌ 高德未安装 → 无降级方案
```

## 2. 信息架构审计

### 2.1 导航结构

```
Tab 栏 (App.vue)
├── 首页 (/) ────────────────────────────── 数据概览
├── 拍照 (/photo) ───────────────────────── 拍照打卡
├── 午餐 (/lunch) ───────────────────────── 午餐转盘
└── 愿望 (/wish) ────────────────────────── 愿望池

隐藏入口
├── 设置 (/settings) ← 首页右上角 ⚙️ 图标
└── 留言管理 (/messages-admin) ← 首页标题连点 5 次
```

### 2.2 问题分析

| 问题 | 描述 |
|-----|------|
| 设置入口不一致 | 首页右上角有 ⚙️ 图标进入设置，但 Tab 栏无设置入口。用户可能找不到设置 |
| 管理页面过于隐蔽 | 连点 5 次标题进入管理页面，无提示。男朋友可能忘记此操作 |
| 层级深度合理 | 所有页面均为一级路由，无深层嵌套，符合移动端最佳实践 |
| Tab 数量适中 | 4 个 Tab 符合移动端 3-5 个的最佳实践 |

### 2.3 路由元信息分析

```javascript
// router/index.js
meta: { title: '首页', icon: '🏠', tab: 0 }      // ✅ 有标题、图标、Tab 索引
meta: { title: '设置', icon: '⚙️', hidden: true } // ⚠️ hidden 但未在 Tab 栏过滤
```

**建议**: Tab 栏应通过 `meta.hidden` 过滤，而非硬编码 4 个 `van-tabbar-item`。

## 5. 无障碍审计

### 5.3 对比度检测

## 6. 首屏体验审计

### 6.1 首页加载流程

```
[进入首页] → [漂浮爱心动画] → [卡片入场动画] → [数字滚动动画] → [数据展示]
     ↓              ↓                ↓                  ↓
  立即渲染      CSS 动画          交错动画            1.2s 滚动
```

### 6.2 首屏元素分析

| 元素 | 加载方式 | 视觉权重 | 问题 |
|-----|---------|---------|------|
| 漂浮爱心 | CSS 动画 | 低（背景装饰） | `Math.random()` 导致位置不稳定 |
| 标题 | 立即渲染 | 高 | 渐变文字效果 |
| 设置图标 | 立即渲染 | 中 | 右上角，可能误触 |
| 数据卡片 | 动画入场 | 高 | 数字滚动需等待 |
| 今日状态 | 动画入场 | 高 | 状态圆圈可点击 |
| 留言预览 | 条件渲染 | 中 | 无留言时不显示 |
| 本周打卡 | 动画入场 | 中 | 进度条有动画 |

### 6.3 首屏问题

| 问题编号 | 描述 | 影响 |
|---------|------|------|
| FS-001 | 漂浮爱心使用 `position: fixed` 覆盖整个页面，可能干扰首屏交互 | 中 |
| FS-002 | 卡片入场动画使用 `--card-delay` 但无 `prefers-reduced-motion` 处理 | 低 |
| FS-003 | 数字滚动动画在 `loveDays` 较大时（如 1000+）仍需 1.2s，用户等待时间长 | 低 |
| FS-004 | 首页无 loading skeleton，数据加载期间显示空白 | 中 |
| FS-005 | 设置入口在首页右上角，与 Tab 栏无设置 Tab 的设计不统一 | 中 |

| 成功提示风格 | `🎉 打卡成功！` (emoji + 文字) | `已删除` (仅文字) |

### 10.2 风格不一致

| 场景 | 可爱风格 | 正式风格 |
|-----|---------|---------|
| 空状态 | `还没有打卡记录哦，开始打卡吧 📸` | `所有数据仅存储在您的设备上...` |

### 10.3 格式不一致

| 问题 | 示例 A | 示例 B |

### 10.4 具体不一致清单

### 11.1 清晰度评估

### 11.2 低清晰度文案

| 文案 | 问题 | 建议 |
|-----|------|------|
| `至少需要 2 家餐厅才能转哟 🥟` | 「转哟」口语化，emoji 与文案无关 | `至少需要 2 家餐厅才能转动哦` |
| `还没有X内容哦` | X 是变量拼接，可能产生「还没有符合条件的哦」 | 拆分为具体文案 |
| `确定删除这条内容吗？` | 「内容」指代不明（愿望/吐槽/留言？） | 明确类型 |
| `谁实现的？(可选)` | 原生 prompt，风格不一致 | 使用自定义弹窗 |
| `该餐厅已存在` | 缺少主语和标点 | `「X」已存在，请修改名称` |
| `没有新的内容可导入` | 信息不全，用户不知道原因 | `导入完成，无新增内容（X 条已存在）` |

## 12. 基于竞品的文案改进方向

### 12.1 竞品分析回顾

根据 `docs/COMPETITIVE_ANALYSIS.md`，本 App 的差异化优势：
- **免登录**: 无需注册，即开即用
- **纯前端**: 数据本地存储，隐私安全
- **AI 彩虹屁**: 智能生成夸赞文案

### 12.2 文案改进建议

#### 12.2.1 强化「免登录」优势

**当前**: 无相关文案
**建议**: 在首次启动时展示引导页

```
标题：「你们的爱情，只属于你们」
副标题：「无需登录，无需网络，所有数据仅保存在这部手机」
按钮：「开始记录我们的故事 💕」
```

#### 12.2.2 强化「纯前端/隐私」优势

**当前**: 隐私说明在设置页底部，用户难以发现
**建议**: 在首页添加隐私徽章

```
文案：「🔒 仅本地存储 · 永不外传」
位置：首页底部或设置入口旁
```

#### 12.2.3 优化「AI 彩虹屁」体验

**当前**: 彩虹屁模板固定，用户无法自定义风格
**建议**:

| 改进点 | 当前 | 建议 |
|-------|------|------|
| 风格选择 | 无 | 添加「夸夸风格」：温柔/搞笑/文艺/土味情话 |
| 频率控制 | 每次打卡随机生成 | 可设置「每日仅一次」或「每次打卡」 |
| 手动刷新 | 无 | 添加「换一句」按钮 |
| 收藏功能 | 无 | 可收藏喜欢的彩虹屁 |

#### 12.2.4 统一文案风格指南

| 维度 | 规范 | 示例 |
|-----|------|------|
| 称呼 | 统一用「宝贝」或用户设置的昵称 | `宝贝，该拍照打卡啦！` |
| 语气 | 可爱但不做作，像男朋友在说话 | `今天的你好好看！` 而非 `这位仙女下凡了呀` |
| emoji | 每句最多 1 个，放在句末 | `🎉 打卡成功！` |
| 标点 | 陈述句用句号，感叹句用感叹 | `已连续打卡 7 天。` vs `太棒了！` |
| 按钮 | 动词 + 名词，可带 emoji | `✨ 添加留言`、`📍 导航` |
| 错误 | 说明原因 + 解决方案 | `无法打开摄像头，请在设置中允许权限` |

#### 12.2.5 建立文案常量文件

**建议**: 创建 `src/constants/messages.js` 统一管理所有文案

export const EMPTY = {
